Повторное развертывание

Произведено повторное развертывание проекта на Railway, в ходе развертывания возниклы проблемы, усеченное содержимое лога:

    ...
    error: rustc 1.85.1 is not supported by the following packages:
    ...
    home@0.5.12 requires rustc 1.88
    icu_collections@2.2.0 requires rustc 1.86
    icu_normalizer@2.2.0 requires rustc 1.86
    ...
  Either upgrade rustc or select compatible dependency versions with
  `cargo update <name>@<current-ver> --precise <compatible-ver>`
  where `<compatible-ver>` is the latest version supporting rustc 1.85.1

По-видимому, Railway по умолчанию использует достаточно старую версию rustc.

В ходе дальнейшего анализа найдены следующие способы решения проблемы:

  1. Создание специального файла rust-toolchain.toml для описания используемой цепочки инструментов: 
    [toolchain]
channel = "<используемая версия Rust>"
components = [<используемые компоненты через запятую>]
targets = ["<архитектура системы>"]
    Информацию для описания можно получить через rustup show и rustup component list.
  2. Понижение версии зависимостей: 
    cargo update <зависимость 1>@<текущая версия> --precise <пониженная версия>
cargo update <зависимость 2>@<текущая версия> --precise <пониженная версия>
...
cargo update <зависимость N>@<текущая версия> --precise <пониженная версия>
    Подходящие версии можно подобрать, непосредственно посетив сайт crates.io и изучив там информацию по проблемным зависимостям.

Понижение версии зависимостей в данном случае не вариант: как мы уже отмечали в начале отчета, Rust и так достаточно увесисто по объему зависимостей тащит, чтобы еще разные версии зависимостей качать. На диске между тем осталось места совсем чуть-чуть...

Поэтому выбран первый вариант: описание в rust-toolchain.toml используемой цепочки инструментов. К моему удивлению, после запуска сборки (cargo build) Rust сделал финт хвостом и начал скачивать и компилировать еще одну цепочку инструментов, идентичную той, что я уже использую. Зачем - мне не понять, да и не вышло ничего, потому что при сборке дублирующей цепочки инструментов закончилось место на диске...

Намучавшись с попытками освободить место в системе, заточенной специально под задачу и не имеющей ничего лишнего помимо инструментов разработки, наконец удалось успешно запустить cargo build и убедиться, что после добавления rust-toolchain.toml проект работает на локальной машине. После публикации на Railway проект также успешно собрался.

Доработка взаимодействия с базой данных

Пока что полноценное взаимодействие с базой данных не налажено. Наброски, описанные в главе База данных, не работают, потому что:

  • не работает извлечение строки подключения из переменных окружения: надо как-то добавить в окружение информацию из файла переменных окружения;
  • не созданы запросы, которые будут извлекать данные из базы и обновлять их.

Решение вопроса разобъем на этапы.

Этап 1: выделение логики, отрабатывающей при успешном получении строки подключения, в отдельную функцию

Так как хочется решать вопрос работы с базой данных независимо от вопроса работы с переменными окружения, поступаю следующим образом.

  1. в проекте создается функция get_application_with_database_support, содержащая обращение к переменной окружения;
  2. в случае успеха эта функция выполняет все действия по инициализации базы данных и возвращает приложение с поддержкой работы с базой данных;
  3. в случае неудачи в основной программе создается урезанная версия приложения без поддержки базы данных, чтобы пользователь мог видеть хотя бы статическое содержимое.

Этап 2: логика для работы с базой данных

  1. в models.rs добавлены модели данных;
  2. в handlers.rs добавлены обработчики для создания нового бота и получения списка ботов;
  3. в main.rs добавлены пути для обработчиков.

Этап 3: подготовка к переезду на Railway

На Railway нужные таблицы сами магическим образом не создадутся, чтобы они были, нужно запустить миграцию.

Добавлена в main.rs строка для запуска миграции:

sqlx::migrate!().run(&pool).await.ok()?;

Этап 4: добавление в окружение информации из файла переменных окружения

В Cargo.toml добавлена зависимость от библиотеки работы с файлами переменных окружения.

В main.rs добавлена строка для инициализации переменных окружения:

dotenvy::dotenv().ok()?;

Этап 5: попытка решения всплывших проблем

Как оказалось, в ветке с поддержкой базы данных была ошибка времени выполнения, просто без инициализации переменных окружения она не выполнялась и ошибки не возникало.

Чтобы разрешить эту проблему, нужна еще одна зависимость для sqlx - runtime-tokio, но добавить её не получается, так как даже после всех чисток и архиваций на диске больше нет места.

Придется отложить решение проблемы на следующую неделю, когда будет время создать новую виртуальную машину с увеличенным объемом жесткого диска.

А пока удаляем снова папку target со скомпилированными зависимостями, чтобы было место хотя бы для элементарных операций вроде сохранения истории работы над проектом.

Исправление проблемы с конечной точкой /api/bots

Изучаю запрос, используемый в данной конечной точке, и что я вижу: SELECT ... FROM users

Но таблицы users в базе данных нет - приложение сгенерировало ошибку при попытке выборки из несуществующей таблицы.

Можно было бы догадаться, что приложение сгенерировало ошибку, если бы добавил в команду curl флаг --verbose, а сама ошибка была видна в другом окне консоли.

Однако ввиду того, что сложно догадаться ставить флаг --verbose и переключиться на другое окно, необходимо менять код.

Меняю код: теперь вместо выброса исключения производится обработка ошибки и в зависимости от того, была ли ошибка или нет, возвращается разный JSON.

В обе версии JSON добавлено поле success: true если ошибки не было, false если была ошибка.

Повторяю тестирование конечной точки /api/bots, и вот результат:

{"success":true,"bots":[{"id":1,"name":"test","token":""}]}
